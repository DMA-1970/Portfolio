Summary: Week 1 introduced the evolution of cloud computing, from early virtualisation technologies to modern multi-cloud ecosystems. It covered the three core service models — IaaS, PaaS, SaaS — and examined deployment models such as public, private, hybrid, and community cloud. Comparisons between AWS, Google Cloud, and Microsoft Azure highlighted service portfolios, scalability options, and pricing structures.
Reflection: As an Enterprise Technology Manager, I had a working knowledge of these foundational concepts, but revisiting them in a comparative context helped me identify subtle provider distinctions that influence strategic decision-making. In my current role, we use AWS S3 buckets for data storage and analytics workloads, while Azure Active Directory underpins enterprise identity management. My cross-platform expertise, supported by certifications such as Microsoft 365 Fundamentals and Microsoft Information Protection and Compliance Administrator Associate, enables me to navigate both ecosystems effectively and ensure security, governance, and compliance alignment. A critical takeaway this week was recognising that while multi-cloud adoption offers resilience and vendor flexibility, it can also create governance challenges and increase the demand for skilled personnel capable of managing multiple platforms.
Application: This understanding directly informs workload placement and procurement decisions, allowing me to balance performance, compliance, and cost efficiency when advising on enterprise cloud strategy.
Forward Planning: I intend to explore further vendor management tools such as AWS Organizations and Azure Lighthouse to strengthen cross-platform governance and streamline policy enforcement across multi-cloud environments.

Coursework: Formative Discussion: ROCCA and TOGAF

Reflective Thinking:

Cloud-Native Tools & Architectures
Summary: This week focused on frameworks that guide cloud architecture. TOGAF offered a structured method for aligning IT with business strategy, the Software Development Life Cycle (SDLC) reinforced disciplined processes in cloud solution delivery, and Infrastructure as Code (IaC) highlighted how automation improves consistency, scalability, and compliance. Collectively, these approaches help ensure cloud deployments are strategic, controlled, and repeatable.
Reflection: TOGAF is particularly relevant because it encourages a business-first approach to cloud architecture. In practice, I have often seen technical teams prioritise tool implementation without aligning outcomes to business needs. TOGAF’s phased Architecture Development Method provides a reminder that strategy and governance should guide deployment choices. SDLC similarly resonated, as cloud environments still require traditional structure for planning, testing, and stakeholder engagement. IaC struck a strong chord with me; having experienced the operational inefficiencies of manual deployments, I see how IaC reduces configuration drift and enforces auditability, which is vital in regulated industries.
Application: Applying TOGAF in my environment could help ensure our AWS and Azure architectures remain strategically aligned and compliant. Embedding SDLC principles into cloud change management would improve testing and rollback, reducing production risk. IaC is already present in Azure deployments for compliance monitoring, but scaling it across both AWS and Azure would improve efficiency while strengthening adherence to frameworks such as GDPR and FCA guidelines.
Forward Planning: I intend to explore how TOGAF ADM can be tailored to hybrid and multi-cloud strategies, and how policy-as-code can complement IaC to improve governance. Embedding SDLC checkpoints into IaC pipelines will also be a priority, ensuring automation remains aligned with compliance and testing requirements.

Summary: This week, we examined hybrid cloud strategies, which integrate on-premises infrastructure with public or private cloud services. We explored patterns such as cloud bursting for peak demand, maintaining data sovereignty for compliance, and segmenting workloads according to security and performance requirements.
Reflection: While I have managed both cloud and on-premises environments in the past, studying structured hybrid models provided a deeper understanding of integration strategies. In my current role, we operate a multi-cloud architecture divided by function: AWS S3 buckets store and manage our data workloads, while Azure Active Directory supports enterprise identity management. Microsoft Purview is used to enforce compliance, and Microsoft Defender safeguards enterprise assets. This arrangement reinforces the need for consistent governance, interoperability, and cross-platform policy alignment — areas where my cross-platform skills and Microsoft certifications have been invaluable. One critical insight is that although multi-cloud enhances resilience and vendor flexibility, it also increases administrative overhead and the need for standardised processes across platforms.
Application: In the financial services sector, hybrid and multi-cloud approaches enable sensitive workloads to remain in controlled environments while using cloud elasticity for analytics and high-volume processing. Our AWS–Azure split not only optimises performance but also mitigates single-vendor dependency and maintains compliance with frameworks such as GDPR and PCI DSS.
Forward Planning: I intend to explore AWS Outposts and Azure Arc to extend hybrid capabilities, enabling unified policy management and more seamless compliance enforcement across both platforms.

Deploying a 12-Factor App to Kubernetes with Minikube (Step-by-Step)

Reflective Thinking:

Summary: This week focused on Infrastructure as Code (IaC) and its importance in cloud environments. TOGAF was introduced as a governance framework for structuring architecture, while IaC enables automation, repeatability, and auditability in deployments. The key activity involved creating and executing a Bash script on OpenStack to automate server setup, reinforcing how automation reduces manual effort and supports scalability. These principles connect closely to Wiggins’ (2017) Twelve-Factor App model, which emphasises consistency and automation in cloud-native design.
Reflection: The scripting exercise was practical and insightful. While I am more experienced with PowerShell and Azure automation, Bash on OpenStack demonstrated the universality of scripting across platforms. IaC emerged as vital for addressing configuration drift, a recurring challenge in multi-cloud operations. The seminar highlighted benefits such as scalability and resilience (Mungoli, 2023), but also the risks of unchecked automation, where errors can propagate rapidly. TOGAF’s phased approach provided a reminder that automation must sit within a strategic governance framework.
Application: In my role, IaC can strengthen compliance by embedding security policies, audit logging, and monitoring directly into code. This is particularly important in financial services, where FCA and GDPR obligations demand consistency. The scripting exercise demonstrated how automation reduces operational overhead while improving standardisation. Oliveira and Hugo (2021) also reminded me that combining IaC with event-driven microservices architectures could further optimise responsiveness and efficiency in cloud deployments.
Forward Planning: I plan to explore policy-as-code tools to embed compliance rules into IaC workflows and extend my scripting skills across OpenStack and multi-cloud platforms. I also aim to connect TOGAF principles with IaC and microservices, ensuring automation is not only efficient but also strategically aligned with enterprise goals.
